# 03 — تحمّل الملايين؟ حكم صادق (بدون مبالغة)

**السؤال:** هل الموبايل + السيستم كبير ويتحمل ملايين مستخدمين/حسابات/إعلانات يومياً؟  
**الجواب الصارم:** **مبني ليتجه للتحجيم، وغير مُثبَت أنه يتحمل الملايين اليوم.**

---

## 1) ما الموجود ويفيد التحجيم

| قدرة | أين | ماذا تعطي |
|------|-----|-----------|
| ترقيم مفتاحي (cursor) للبحث | `SearchService.ts` | صفحات نتائج بدون سحب الجدول كله |
| تجميع خريطة من السيرفر | `mapClusters` + `/search/map` | لا تُرسل كل الدبابيس |
| فهارس + GIN trigram | schema + bootstrap | بحث نصي أسرع من ILIKE أعزل |
| فهارس فئة/حالة/سعر/feed | `lib/db` schema | فلاتر أساسية |
| Rate limit | `rateLimiter.ts` + AbuseService | حماية سوء استخدام |
| تخزين كائنات (S3 path) | object storage | وسائط خارج عملية Node |
| دورة نشر مختبرة | MarketplaceLifecycle e2e | صحة وظيفية لا حمل |

---

## 2) ما الناقص لإثبات «ملايين يومياً»

| فجوة | الأثر |
|------|--------|
| لا اختبار حمل/soak موثّق (k6 وغيرها) | أي رقم ملايين = تخمين |
| Rate limit / بعض الـ cache داخل العملية | تعدد instances يضعف الحماية |
| لا Redis مشترك / لا طابور رسائل حقيقي | إشعارات وjobs تحت ضغط هشة |
| البحث ما زال نموذج SQL+trigram | عند عشرات الملايين من الصفوف قد يلزم محرك بحث |
| لا read replica موثّق في التشغيل | قراءة البحث تنافس الكتابة |
| Observability جزئي (لا Sentry/مقاييس منتج كاملة) | لا نرى الانهيار مبكراً |
| Replit ≠ إنتاج عالمي | بيئة مختلفة عن AWS/GCP كامل |

---

## 3) حكم الطبقات

| الطبقة | جاهزية وظيفية | جاهزية ملايين |
|--------|----------------|----------------|
| Publish lifecycle | عالية (مختبرة) | غير مُثبتة تحت حمل |
| Search/map | جيدة معمارياً | تحتاج حمل + replicas محتملة |
| Auth (Clerk) | يعتمد مزوداً خارجياً | قابل للتحجيم إن ضُبط |
| Mobile UX | غني | العميل خفيف نسبياً؛ العنق = API/DB |
| FI/Banks | جزئي | لا علاقة له بالملايين قبل إغلاق الحلقة |
| Admin | تشغيلي | ليس مسار المستخدم العام |

---

## 4) ماذا تعني «يتحمل» عندنا (تعريف قبول لاحق)

لا نكتب «يدعم ملايين» في تسويق قبل:

1. سيناريو حمل مكتوب (مثلاً: X بحث/ث · Y نشر/دقيقة · Z رفع صورة)  
2. تشغيل على staging قريب من الإنتاج  
3. p95 لـ `/search` و `/feed` تحت العتبة المتفق عليها  
4. خطأ < عتبة · لا انهيار DB  
5. خطة dig عند الفشل (replica / الموجة E في الشيت 02)

---

## 5) قرار مطلوب منك

| خيار | المعنى |
|------|--------|
| ☐ نؤجّل ادعاء الملايين | نكمل صحة المنتج + Ops أولاً (موصى به) |
| ☐ نفتح موجة Scale الآن | بعد اعتماد E1–E7 جزئياً وميزانية بيئة |

**توصية الخطة:** أكمل موجات Ops + فصل + FI الأساسية، ثم موجة Scale بإثبات — لا توسع عشوائي «عشان الملايين».
