## 0) إجابة سؤالك الأول: ليه المشاكل بترجع بعد الإصلاح / بعد الانتقال؟

### الآلية الجذرية (مثبتة بالجيت — مش رأي)

```
إصلاح جراحي دقيق على main
        ↓
كومِت وكيل ريبلِت ضخم بعنوان ضيق (مثل Auto-seed)
        ↓
يعيد كتابة عشرات/مئات الملفات نحو شكل الـbaseline القديم
        ↓
الميزة «تختفي» رغم إن الإصلاح كان موجود
        ↓
استعادة لاحقة جزئية (ترجع فيتشرات كبيرة وتنسى UX الجراحي)
        ↓
كومِت بعنوان «menu fix» يحرّك الزر فقط → يوهم إن المشكلة اتحلّت
        ↓
نفس العرض القديم يرجع (أنت شفته بعينك)
```

## 2) قائمة تشريح المشاكل (Anatomy List)

### P-01 — قائمة البروفايل تملأ الشاشة ولا تتقفل صح
| | |
|--|--|
| **العَرَض (مالك)** | قائمة بحجم الشاشة · صعوبة إغلاق · رجعت بعد إصلاح دقيق |
| **السبب الجذري** | رجوع نمط baseline بعد `93b650b` + غياب `maxHeight`/`ScrollView` |
| **طبقة** | CODE regression (تلوث seed عبر ريبلِت) |
| **ليس السبب** | Clerk · قيم ENV · ريبو b.deals |
| **الإصلاح الآمن لاحقاً** | إعادة **فقط** قالب Modal من `f70e016`+`4ccf939` (backdrop شقيق + scroll/cap) · لا rewrite للبروفايل |
| **تبعيات** | `PromoteButton` كان له نفس مرض اللمس في نفس إصلاح `f70e016` — افحصه عند التنفيذ |
| **اختبار قبول** | Android+iOS: فتح ⋯ · ضغط صف · غلق بالخلفية · قائمة طويلة لا تتجاوز الشاشة · testIDs `menu-*` |
| **حارس ضد الرجوع** | اختبار ثابت يفشل لو وُجد `onStartShouldSetResponder` داخل menu Modal أو غاب `maxHeight` |

### P-02 — رفع الصور/الفيديو / السيرفر / الأسلاك
| | |
|--|--|
| **المسار الحي** | `create.tsx` → `lib/upload.ts` → API uploads → storage PUT → verify → عند النشر `ListingService` + promote |
| **كود ميت** | `ListingMediaEditor.tsx` بلا أي import — خطر على أي «إصلاح» يستهدفه بالغلط |
| **أمن** | `upload_claims` + `assertCallerMayUseUpload` (C-01 / `e24014b`) + اختبار IDOR موجود |
| **كسر تشغيلي مثبت في الكود** | رسالة 503 «storage unconfigured» من `0afef07` **اتمسحت في `93b650b`** → HEAD يرجع 500 غامض لو المسارات ناقصة |
| **edit listing** | `edit/[id].tsx` حالياً لا يعيد رفع وسائط (سعر/سوق…) — wave 10C كان في تاريخ؛ لا تفترض edit-media حي بدون التحقق وقت التنفيذ |
| **طبقة المشاكل المحتملة في برودكشن حي** | (1) مسار العميل vs مسار ميت (2) فشل PUT/ACL (3) claim منتهي (4) serve URL (5) تلوث بعد wipe — **التشخيص يلزم سجلات request-id من السيرفر الحي**، مش افتراض |
| **خطة فحص قبل لمس كود** | Trace طلب واحد حي: request-url → PUT status → verify → create listing → GET object URL من الفيد |

## 3) ليه «مكانش عندي المشاكل دي وكان كله شغال»؟

ثلاث طبقات متزامنة (كلها مثبتة أو موثّقة في الجيت):

1. **كان شغال** بعد الإصلاحات الدقيقة (`f70e016`/`4ccf939`/طبقة نوتفكيشن ناضجة).  
2. **اتمسحت** بكمّات ريبلِت واسعة النطاق (`93b650b` وأمثالها / sync era).  
3. **الاستعادة ناقصة أو تجميلية** → العرض القديم رجع وأنت صدّقت عينك.

هذا **ليس** تناقض مع إن Clerk والبرودكشن مفعّلين. التفعيل الخارجي لا يمنع **تلوث شجرة المصدر**.

## 4) تقنيات الإصلاح الآمن (قبل التنفيذ — قواعد صارمة)

1. **Surgical restore فقط** — انقل قالب الإصلاح من SHA معروف، لا «أعد كتابة الميزة».  
2. **حارس انحدار (regression guard)** لكل إصلاح جراحي (مثل اختبار قائمة البروفايل).  
3. **منع كومِتات ريبلِت واسعة** — أي تغيير > نطاق المهمة يُرفض؛ فرق الملفات يُراجع سطراً.  
4. **لا تستهدف كود ميت** (`ListingMediaEditor`) قبل إثبات الاستيراد.  
5. **لا تخلط ENV مع CODE في نفس الـPR** عند التشخيص — لكن البرودكشن عندك حي: التشخيص يبدأ بـ**سجلات وSHA** لا بافتراض أسرار ناقصة.  
6. **فرع واحد لكل محور** · merge بعد typecheck + حارس + قبول مالك على جهاز.  
7. **ممنوع merge** فروع موثّقة كخطر (مثل booking-notif contract) بلا مراجعة عقد.  
8. **مرايا** (`b.deals`/`B-OOM`/`aws-virgen`) = أقدم من CA-OOM؛ ليست مصدر استعادة أحدث.

## 5) خطة برودكشن كاملة (مراحل — بدون تنفيذ الآن)

### المرحلة 0 — تجميد الضرر (Policy)
- إيقاف أي Agent ريبلِت يلمس الموبايل بعناوين عامة.  
- أي صيانة: قائمة ملفات مسموحة مسبقاً.  
- توثيق SHA البرودكشن الحي يومياً.

### المرحلة 1 — Truth pack (قياس بلا كود)
| فحص | الغرض |
|------|--------|
| SHA المنشور vs `origin/main` | هل العرض من كود قديم؟ |
| سيناريو بروفايل ⋯ على ASB/جهاز | تأكيد P-01 حي |
| Trace رفع واحد (status codes) | تأكيد أين ينقطع P-02 |
| `/search/map` + خريطة native | P-03 |
| صف `push_tokens` + نوع البناء (Go vs ASB) | P-05 |
| قرار مالك مكتوب: Facebook Login نعم/لا | P-04 |

### المرحلة 2 — إصلاحات جراحية بالترتيب (بعد موافقة)
1. **P-01 بروفايل Modal** (أقل تبعيات · أعلى يقين · يطابق عين المالك)  
2. حارس انحدار للقائمة  
3. **P-05** تفريق in-app vs push vs email حسب نتائج المرحلة 1  
4. **P-02** حسب نقطة القطع في الـtrace فقط  
5. **P-03** حسب نقطة القطع فقط  
6. **P-04** بعد قرار Facebook فقط  
7. **P-06** قبل أي ASB جديد إن اللوم على الأيقونة

### المرحلة 3 — تحصين
- حراس: section melt · profile menu · i18n · upload IDOR (موجود)  
- منع commit لـ `dist`/`static-build`  
- بوابة CI ترفض إعادة `onStartShouldSetResponder` في menu Modal

### المرحلة 4 — مزامنة مرايا النشر
- بعد استقرار `main` فقط: `bancooom` / `aws-virgen` بأدوات النشر الرسمية — **ليست** مصدر إصلاح.

---

## 6) ما لن أفعله الآن (التزام)

- لن أعدّل `profile.tsx` أو uploads أو notifications أو Clerk في هذه الجولة.  
- لن أدمج مرايا.  
- لن أفترض إن المفاتيح ناقصة رغم إقرارك إن البرودكشن حي.  
- التشخيص القادم لأي عَرَض حي يبدأ بـ: **SHA + أثر + مسار سلك + تبعيات**.

## 7) ملخص تنفيذي

| سؤالك | الجواب المثبت |
|-------|----------------|
| ليه بترجع؟ | كومِتات ريبلِت ضخمة تعيد baseline + استعادة ناقصة |
| ليه البروفايل رجع؟ | `93b650b` مسح `f70e016`+`4ccf939` · لسه في HEAD |
| ليه النوتفكيشنز؟ | طبقة اتمسحت (موثّق) + قيود Expo Go/ASB + تفضيلات |
| هل المشكلة تعريف؟ | نعم جزئياً (Facebook في الكود غايب رغم التفعيل) · وأيضاً تلوث مصدر |
| منين نستعيد؟ | من تاريخ **نفس** `-BANCO-CA-OOM-` بـSHAs معروفة — لا من b.deals/B-OOM |

**الخطوة التالية بعد موافقتك:** تنفيذ المرحلة 1 (Truth pack) فقط، ثم P-01 جراحياً مع حارس — بعد ما تعتمد قائمة الملفات المسموحة.
