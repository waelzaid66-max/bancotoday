# @clerk/expo + Expo Go: pin to 3.3.1

**Rule:** For an Expo app that runs in **Expo Go** (workflow runs `expo start`, no custom dev build), pin `@clerk/expo` to exact `3.3.1`. Do NOT let it float (caret `^`) and do NOT "update to latest" (3.4.x/3.5.x) for Expo Go.

**Why:** Starting at `3.4.0`, `@clerk/expo` ships `dist/specs/NativeClerkModule.android.js` that calls `requireNativeModule("ClerkExpo")` at module **top-level**, and `dist/utils/native-module.js` top-level-`require`s that spec **outside** any try/catch. In Expo Go the `ClerkExpo` native module isn't registered, so the require **throws at module-load time** → uncaught `Cannot find native module 'ClerkExpo'`. The `loadNativeModule()` try/catch guard never runs because the throw happens during the import graph evaluation. **Android only** — iOS/web resolve to `NativeClerkModule.js` (`TurboModuleRegistry.get` → null) / `NativeClerkModule.web.js`, which are safe. `3.5.x` added a guarded `requireNativeModule` fallback inside `loadNativeModule`, but it STILL top-level-requires the throwing android spec, so it does NOT fix Expo Go. `3.3.1` ships **no** `NativeClerkModule.android.js` at all, so Android resolves to the safe base spec — and it still exports every API the app uses (`useSSO`, `useSignInWithGoogle`, `useOAuth`, `useSignIn`, `useSignUp`, `@clerk/expo/token-cache`).

**How to apply:** If the user reports `Cannot find native module 'ClerkExpo'` (or any Clerk native-module crash) in Expo Go on Android: verify the installed version (`node_modules/@clerk/expo/package.json`) and the presence of `dist/specs/NativeClerkModule.android.js`. If the android spec exists, the version is too new for Expo Go — pin `@clerk/expo` to exact `3.3.1` in the mobile artifact's `package.json` (exact pin also bypasses `minimumReleaseAge`), `pnpm install`, restart the expo workflow to clear Metro cache. This is NOT code divergence — the app's `_layout.tsx` ClerkProvider wiring can match the skill canonical exactly and still crash purely from the version float. The clerk-auth skill's "update @clerk/* to latest" troubleshooting step is wrong for Expo Go and reintroduces the crash. To verify the fix without an Android device, the headless screenshot of an RN-web Expo app is often blank (capture quirk), so confirm via the bundle logs (no ClerkExpo error / no redbox) and ask the user to reload Expo Go on a real Android device.
