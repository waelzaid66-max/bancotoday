# BANCO Abuse Control & Revenue Protection

The business-layer abuse/revenue-protection system. Rules below are durable invariants, not implementation logs.

## Layering invariant
- All abuse decisions live in `AbuseService` (in-memory sliding-window primitives from `lib/slidingWindow.ts`) + persisted state in DB. Services call into AbuseService; they do NOT re-implement detection inline.
- Every abuse action (blocked lead, suspicious click, invalid impression, flagged listing, price outlier, spam, rate limit, shadow ban) MUST be recorded via `AbuseService.writeAudit` to the durable `audit_log` table. If you add a new abuse path, add an audit write too.
- **Listing flags specifically:** setting `isFlagged/flagReason` on a listing is NOT enough — you must also call `AbuseService.auditListingFlag(...)` so the flag lands in `audit_log` (emits `spam_content`/`price_outlier` + a unified `flagged_listing` marker). Wired in BOTH `ListingService.createListing` and `updateListing`. **Why:** a code review rejected an earlier version that persisted the flag column but never wrote the audit trail.

## Visibility invariant (critical)
- `lib/feedVisibility.ts` `publicVisibilityConditions()` returns the canonical "hide flagged listings + shadow-banned sellers" SQL conditions.
- EVERY public listing-surfacing query must spread these conditions in its WHERE: feed, search, autocomplete, trending, similar, public listings, active ads, company-profile listings, cross-listing supply-chain links. Any query that filters `users` for shadow-ban needs the `users` join present (autocomplete needed a join added for this).
- **Aggregate/COUNT queries count too.** A public stats surface (e.g. company profile `active_listings`/`total_listings`) that does `COUNT(*) FROM listings WHERE userId=...` without the `users` join + `publicVisibilityConditions()` will leak flagged-listing counts. A code review flagged exactly this. Also 404 the whole public profile when the seller is shadow-banned (return null) — don't expose identity/stats for a suppressed seller.
- **New standalone entity types that mirror listing visibility (investments, global-supply requests) need the gate replicated in THREE places, not one:** (1) the public list/browse WHERE; (2) the `getById` detail — the list filter does NOT protect direct-id fetches, so non-owners must be re-gated on `isFlagged`/owner `isShadowBanned` after the row loads (owner still sees their own); (3) any cross-entity aggregate that joins back to `listings` (e.g. market-intelligence lead/demand counts join `leadHistory`→`listings`→`users` and must carry `listings.isFlagged IS NOT TRUE` + `users.isShadowBanned IS NOT TRUE`, matching the sibling listing aggregate). A code review caught all three missing on the first pass of the B2B surfaces. Also gate the write/interest path (submit-interest, respond) so outsiders can't act on a suppressed item.
- **Relationship/contact creation is a surface too, not just reads.** `ConversationService.createConversation` (the buyer "Contact seller" path) fetches the listing by id and MUST gate `eq(status,"active")` + `users` join + `...publicVisibilityConditions()` — otherwise a known listing UUID lets anyone start a thread with the seller of withdrawn/flagged/shadow-banned inventory even though `getListingDetail` 404s that same listing to non-owners. Gating creation is safe: the contact button is only reachable from an active detail page, and existing threads are reopened via the inbox (`loadParticipantConversation`/getMessages), which is unaffected. A code review caught this missed on the first stabilization pass.
- **Billable/contact event tracking is a surface too.** `LeadService.trackLead` (contact → lead row + seller notify + CPL charge) MUST gate its listing lookup on `eq(status,"active")` + `users` join + `...publicVisibilityConditions()`, exactly like `createConversation`. Otherwise a harvested listing UUID forges billable leads against withdrawn/flagged/sold/shadow-banned inventory (drains CPL budgets, spams sellers). Non-public listing → no row → the fire-and-forget path silently returns before insert/notify/charge.
- **A user report must NOT auto-change listing status.** `ReportService.createReport` records the report as `open` for the admin queue (`listReports` + the moderation queue's open-report `exists` predicate); it does NOT flip the listing to `pending_review`. **Why:** one report from any authenticated user instantly flipping active→pending_review hid the listing from feed/search/detail = competitor-takedown / DoS. Only an explicit admin action (`AdminService.moderateListing`) may hide a listing. The report route uses `requireDbUser` (accountable reporter) + dedup: one OPEN report per (reporter, listing). **The dedup is enforced at the DB by a partial unique index `uq_reports_open_reporter_listing` ON (listing_id, reporter_user_id) WHERE status='open' — that index is the source of truth.** The app-level precheck SELECT can race, so `createReport` wraps the insert in try/catch and on `isUniqueViolation` re-queries and returns the existing open report instead of surfacing a 500 (fail-closed: rethrow if the re-query finds nothing). Index allows duplicate NULL reporters, but the route always sets one.
- **Why:** shadow ban is silent — the banned user sees their own content normally; only public surfaces hide it. Missing the condition on a new surface (including count/stat surfaces) leaks shadow-banned/flagged content.

## Revenue invariants
- Fraudulent/duplicate/bot leads are validated BEFORE insert and never charged. Lead dedup window is short (<60s repeat click = blocked).
- Ad impressions: raw `impressions` counter always advances (analytics), but only valid-session impressions are billable → increment `billable_impressions` + deduct `cost_per_impression` from `budget_total`; ad auto-deactivates when spend reaches budget. Bot/invalid impressions never drain budget.
- **Impression billing must NOT trust a bare `session_id` string.** A billable impression requires an authenticated user OR a session the server itself created (`AdaptiveFeedEngine.hasSession`, populated by feed browsing / behavior signals). Plus a hard per-IP billable cap so an attacker can't bypass per-session limits by rotating fabricated session ids from one origin. **Why:** a code review rejected a version where any non-empty `session_id` was billable → trivial budget-drain attack. **How to apply:** any new billable/credit-consuming endpoint keyed on a client-supplied id must verify that id against server-side state + add a per-IP cap.

## FeedItem contract still immutable
- Abuse work only changes trust_signal/smart_badge VALUES (added "Top Dealer" = verified + dealer/company/enterprise + qualityScore>=80; "High Demand" = clicks>50). No new FeedItem fields. quality_score flows through the row pipeline (BffService.RawListingRow.quality_score optional, enrichListings spreads `...row`) but is NOT a FeedItem field — it only feeds ranking + trust signal.

## Gotchas
- `ads` table has `createdAt` but NO `updatedAt` — do not set updatedAt in ads updates.
- error code enum has no RATE_LIMIT; rate-limit responses use HTTP 429 with code "INVALID_DATA". Services throw `{code:"RATE_LIMITED"}`, controllers map to 429.
- Comparing an enum column (e.g. `listingAttributes.propertyType`) against a plain string needs a cast to `typeof col.enumValues[number]`.
- Dealer quality recompute is fire-and-forget (not awaited) after listing create / lead-status change; DEALER_ROLES = dealer/company/enterprise.
